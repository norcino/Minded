Recursion Prevention in Decorators Using IMindedContext Scoped Values

Problem

A command or query handler decorator may need to dispatch its own mediator calls to fulfil its cross-cutting responsibility (e.g., an authorization decorator invoking a GetUserPermissionsQuery or a CheckResourceOwnershipCommand). Because those nested calls re-enter the decorator pipeline, the same decorator will be invoked again for the nested call, which re-dispatches its own mediator call, and so on indefinitely.

Mechanism

IMindedContext provides a per-logical-call scoped stack via:

IDisposable BeginScope<T>(T value);
bool TryGetScoped<T>(out T value);

A value pushed with BeginScope<T> flows forward into the current async continuation and into any nested mediator calls dispatched from it, but is not visible to sibling branches that were forked before the scope was opened. When the returned handle is disposed the value is removed from the current flow.

This asymmetry — forward-propagating, parallel-isolated — is exactly what a recursion guard needs: the nested call sees the flag; anything else running concurrently does not.

Contract for the Decorator Author

Step 1 — Declare a marker type

Define a private or internal readonly struct in the decorator (or its assembly). No members are needed; the type itself is the key.

private readonly struct AuthorizationBypass { }

A struct is preferred over a class: no allocation, no inheritance, value-equality by default.

Step 2 — Check the scope at the top of the decorator

Before doing any work, ask the context whether the current call was dispatched from inside a bypass scope. If yes, skip the decorator's logic entirely and delegate straight to the next handler.

if (_contextAccessor.Current.TryGetScoped<AuthorizationBypass>(out _))
    return await _next.HandleAsync(command, cancellationToken);

Step 3 — Open the scope only around the nested mediator call

When the decorator dispatches its own mediator call, wrap that call — and only that call — in a BeginScope. Dispose the scope before delegating to the real next handler so that the business logic that follows runs with full decoration.

using (context.BeginScope(new AuthorizationBypass()))
{
    var result = await context.Mediator.ProcessCommandAsync(permissionCheckCommand, cancellationToken);
    if (!result.Successful)
        return Forbidden<TResult>();
}

// Scope is gone. Real handler and any commands it dispatches are fully decorated.
return await _next.HandleAsync(command, cancellationToken);

Step 4 — Do not hold the scope open past the guard check

If the scope is still open when _next.HandleAsync is called, any sub-call the handler makes will also skip the decorator. Closing the scope before delegating preserves full decoration for legitimate business commands.

Full Pattern

public class AuthorizationCommandHandlerDecorator<TCommand, TResult>
    : ICommandHandler<TCommand, TResult>
    where TCommand : ICommand<TResult>
{
    private readonly ICommandHandler<TCommand, TResult> _next;
    private readonly IMindedContextAccessor _contextAccessor;
    private readonly IPolicyEvaluator _policies;

    private readonly struct AuthorizationBypass { }

    public async Task<ICommandResponse<TResult>> HandleAsync(
        TCommand command, CancellationToken cancellationToken = default)
    {
        var context = _contextAccessor.Current;

        // Guard: this call was dispatched from inside a policy check — skip.
        if (context.TryGetScoped<AuthorizationBypass>(out _))
            return await _next.HandleAsync(command, cancellationToken);

        // Evaluate policy. The scope prevents re-entry for this subtree only.
        using (context.BeginScope(new AuthorizationBypass()))
        {
            var allowed = await _policies.EvaluateAsync(command, context, cancellationToken);
            if (!allowed)
                return CommandResponse<TResult>.Forbidden();
        }

        // Scope closed. Business handler receives full decoration.
        return await _next.HandleAsync(command, cancellationToken);
    }
}

Rules

RuleReason
Check the scope at the very start, before any logicEnsures the guard fires even when there is no policy attribute on the command
Open the scope only around the internal mediator call, never around _next.HandleAsyncPrevents the business handler's sub-commands from inheriting the bypass
Declare the marker type as a private readonly structKeeps the key unique and allocation-free; cannot be accidentally reused from outside the decorator
Use one marker type per decorator responsibilityTwo decorators with distinct marker types can each guard their own recursion independently
Never read or write the marker from outside the decorator that defined itIt is an implementation detail, not a public contract

Parallel Safety

If a handler dispatches two commands concurrently via Task.WhenAll, each branch enters the authorization decorator independently. A bypass scope opened inside one branch's policy check is not visible to the other branch because both branches forked before the scope was opened. Each branch therefore independently passes or fails authorization — no cross-contamination.
